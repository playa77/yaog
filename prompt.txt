### **Deliverable 1: Design Document (v1.1)**

**Project Title:** `OR-Client` - A Standalone OpenRouter.ai Frontend

**Version:** 1.1

**Date:** November 13, 2025

---

#### **1. Project Overview & Philosophy**

`OR-Client` is a standalone desktop application for Ubuntu Linux designed to provide a power-user interface for the OpenRouter.ai API. Its core philosophy is to offer maximum, per-generation control over model parameters in a fast, native, and uncluttered environment. The user experience is inspired by the clean, three-panel layout of Google AI Studio, prioritizing workflow efficiency over broad, non-essential features. All data, including conversation history and custom personas, is stored locally, ensuring user privacy and offline access to past chats.

#### **2. Functional Requirements (FR)**

*   **FR1: Granular, Per-Generation Model Control:** At any point in a conversation, the user can change the model, temperature, and system prompt. These new settings will apply *only* to the next LLM generation. The conversation history will visually indicate which model and settings produced each response.
*   **FR2: Comprehensive Chat Interface:**
    *   A central message display area for rendering conversation history, including formatted code blocks and Markdown.
    *   A text input area for user prompts.
    *   A real-time token counter for the current context.
*   **FR3: Conversation History Management:**
    *   All conversations are saved automatically.
    *   A dedicated panel lists all past conversations, identified by a title (e.g., the first user prompt).
    *   Users can select a conversation to load it into the main chat window.
    *   Users can search conversations by content.
    *   Users can add/remove tags to/from conversations for organization.
*   **FR4: System Prompt / Persona Management:**
    *   A dropdown lists available system prompts (e.g., "Python Expert," "Creative Writer").
    *   Users can select, view, edit, and save their own custom system prompts. These are persisted locally.
*   **FR5: File Attachment Support (Client-Side Extraction):**
    *   Users can attach `.txt` and `.pdf` files to a prompt.
    *   The application will extract the text content from the file and prepend it to the user's message text before sending the API request.
*   **FR6: Web Search Toggle:**
    *   A simple UI toggle (e.g., a checkbox) allows the user to enable web search for a query.
    *   This control passes the required boolean flag to the OpenRouter API.
*   **FR7: API and Model Configuration:**
    *   The application loads the user's OpenRouter API key from a local `.env` file.
    *   The list of available models is loaded from a local configuration file (`models.json`) at startup for speed and offline availability.
*   **FR8: Copy Functionality:** Users must be able to easily copy the content of any assistant-generated message to the clipboard via a visible "Copy" button or icon associated with each message block.

#### **3. Non-Functional Requirements (NFR)**

*   **NFR1: Platform:** The application is exclusively for Ubuntu Linux.
*   **NFR2: Performance:** The UI must remain responsive at all times. API calls are asynchronous to prevent freezing. Startup time should be minimal.
*   **NFR3: Data Storage:** All user data (history, prompts, tags) will be stored in a single, local SQLite database file located in the user's configuration directory.
*   **NFR4: Security:** The API key is managed by the user in an external `.env` file. The application reads it but does not expose it in the UI or store it elsewhere.
*   **NFR5: Usability:** The UI will adhere to a strict three-panel layout: History (left), Chat (center), and Controls (right).

#### **4. Technology Stack**

*   **Language:** Python 3.10+
*   **UI Framework:** PyQt6
*   **Chat Rendering:** `QWebEngineView` to ensure high-fidelity rendering of Markdown and code snippets.
*   **Data Storage:** Python's built-in `sqlite3` module.
*   **API Communication:** `httpx` for asynchronous API calls.
*   **Configuration:** `python-dotenv` for loading the API key.
*   **PDF Parsing:** `PyMuPDF` for reliable text extraction from PDF files.

#### **5. Architecture & UI Layout**

*   **Main Window (`QMainWindow`):**
    *   **Left Dock (`QDockWidget`): History Panel**
        *   `QLineEdit` for search functionality.
        *   `QListView` to display conversation titles and tags, backed by a `QAbstractListModel`.
    *   **Center Widget: Chat Panel**
        *   `QLabel` at the top for the token count.
        *   `QWebEngineView` taking up the majority of the space. The HTML content will be dynamically generated and will include CSS and JavaScript to render message blocks. Each assistant message block will contain a "Copy" icon.
        *   `QTextEdit` at the bottom for user input.
    *   **Right Dock (`QDockWidget`): Controls Panel**
        *   `QComboBox` for Model Selection.
        *   `QLabel` + `QSlider` for Temperature.
        *   `QLabel` + `QSpinBox`/`QSlider` for Token Limit.
        *   `QComboBox` for System Prompt selection.
        *   `QTextEdit` to display/edit the selected system prompt.
        *   `QPushButton` to "Save" changes to a system prompt.
        *   `QCheckBox` for "Enable Web Search".

*   **Data Model (SQLite Schema):**
    *   `conversations` table: `id` (PK), `title` (TEXT), `created_at` (TIMESTAMP).
    *   `messages` table: `id` (PK), `conversation_id` (FK), `role` (TEXT, 'user' or 'assistant'), `content` (TEXT), `model_used` (TEXT), `temperature_used` (REAL), `timestamp` (TIMESTAMP).
    *   `system_prompts` table: `id` (PK), `name` (TEXT UNIQUE), `prompt_text` (TEXT).
    *   `tags` table: `id` (PK), `name` (TEXT UNIQUE).
    *   `conversation_tags` table: `conversation_id` (FK), `tag_id` (FK).

*   **Core Logic:**
    *   **`APIManager` Class:** A dedicated module to handle all interactions with the OpenRouter API. It will be responsible for constructing headers, formatting the request body (including the message history), and parsing the JSON response. It will use `httpx` to make non-blocking calls.
    *   **`DatabaseManager` Class:** A module to abstract all SQLite operations (CRUD for all tables).
    *   **`MainWindow` Class:** The main application class that connects UI signals to slots. It will also use a `QWebChannel` to bridge communication between the Python backend (e.g., for clipboard access) and JavaScript running inside the `QWebEngineView` (for the copy button).

    4.  Parse the API response and append both the user's prompt and the assistant's response to the `QWebEngineView`.
